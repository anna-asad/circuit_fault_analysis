## 1. Project Overview

This is a browser-based circuit fault detector. A user draws a circuit on an interactive canvas, clicks **Simulate**, and the system:

1. Converts the drawn graph into a SPICE netlist
2. Runs the netlist through **ngspice** for a DC operating-point analysis
3. Parses node voltages and branch currents from ngspice's output
4. Runs **two independent fault detectors** in parallel:
   - A rule-based **structural fault detector** (open circuits, shorts, reversed polarity, etc.)
   - A trained **RandomForest ML model** that classifies component-level fault types
5. Returns all results to the browser and displays them in a results panel

The ML model was trained on synthetic circuit data generated by running thousands of intentionally-faulted circuits through ngspice.

## 3. Frontend — Circuit Editor

### 3.1 App Shell & State (`App.jsx`)

`App.jsx` is the top-level component. It owns all shared state and routes between the editor view and the results page.

**State held at app level:**

| State variable | Type | Purpose |
|---|---|---|
| `circuit` | `{ nodes[], edges[] }` | Live ReactFlow graph, updated on every canvas change |
| `simulationResults` | object or null | Full API response from `/api/simulate` |
| `isSimulating` | boolean | Disables the simulate button during a request |
| `showResultsPage` | boolean | Switches between editor and full-screen results |

**Page routing:** The app does not use React Router. It uses a ternary — when `showResultsPage` is true, the entire app renders `ResultsPage` (wrapped in a `100vw × 100vh` div to prevent height collapse). When false, it renders the normal editor layout.


### 3.2 Component Sidebar (`ComponentSidebar.jsx`)

A fixed-width left panel listing 6 draggable component types:

| Type | Label | Default Value |
|---|---|---|
| `dc_source` | DC Source | 5 V |
| `resistor` | Resistor | 1000 Ω |
| `capacitor` | Capacitor | 100 nF (1e-7 F) |
| `inductor` | Inductor | 1 µH (1e-6 H) |
| `junction` | Junction | — |
| `ground` | Ground | — |

Each item uses the HTML5 drag-and-drop API. On `dragStart` it writes the component type string into `event.dataTransfer` under the key `application/reactflow`. The canvas reads this on `drop`.

### 3.3 Circuit Canvas (`CircuitCanvas.jsx`)

The main editing area. Built on **ReactFlow v11**.

**Node types registered:**

- `componentNode` — renders a component SVG (resistor zigzag, capacitor plates, inductor arcs, battery symbol) with a clickable value label below it
- `junctionNode` — small filled dot, used as a T/cross connection point
- `groundNode` — three-bar ground symbol

**Capabilities:**

- **Drag to place** — dropping a component from the sidebar creates a new ReactFlow node at the drop position using `instance.screenToFlowPosition()`. The ReactFlow instance is stored in a `ref` (not just state) to avoid stale closures on drop.
- **Wires** — dragging from a handle creates a `straight` edge. Handles are `type="source"` on both ends (connection mode is `"loose"` so either end can initiate).
- **Inline value editor** — clicking the underlined value label below a component opens a small inline input field directly on the node (no browser `prompt()`). Pressing Enter or clicking ✓ saves; Escape or ✕ cancels.
- **Delete** — selecting a node or edge and pressing `Delete` or `Backspace` removes it.
- **Read-only mode** — when `mode="results"`, dragging, connecting, and deleting are all disabled. Used in the results page canvas replay.
- **Sync to parent** — `useEffect` calls `setCircuit({ nodes, edges })` whenever the canvas state changes so `App.jsx` always has the current graph.

**SVG symbols** are defined inline in the component as `COMPONENT_SVGS` — a map of component type → JSX SVG element. These use `currentColor` for strokes so they inherit the node's text color.


### 3.4 Circuit-to-Netlist Converter (`circuitConverter.js`)

This is the most algorithmically complex part of the frontend. It converts the ReactFlow graph (which is a visual representation with pixels and node IDs) into the electrical connectivity model the backend expects.

**Input:** `{ nodes[], edges[] }` — raw ReactFlow state  
**Output:** `{ nodes: string[], components: ComponentModel[], ground: "0" }`

**Algorithm — Union-Find node assignment:**

1. **Classify nodes** into `componentNodes`, `junctionNodes`, and `groundNodes`.
2. **Build adjacency graph** — for each edge, push neighbor IDs into both endpoint's lists.
3. **Handle-level validation** — track `nodeId::handleId` pairs; throw if any component pin has more than one wire attached.
4. **Connection count validation** — each component must have exactly 2 non-ground wire connections. Counted by raw array length (not distinct neighbors), so two wires from the same component to the same neighbor (a direct loop) correctly counts as 2.
5. **Union-Find setup** — create entries for all ReactFlow node IDs plus synthetic terminal IDs (`compId_t0`, `compId_t1`) for each component's two pins.
6. **Process edges** — for each wire, union the appropriate terminals:
   - Component ↔ Component: `claimNeighborIndex()` assigns the first wire to t0, second to t1
   - Component ↔ Junction: union the component terminal with the junction node
   - Ground ↔ anything: union with the ground group
7. **Ground proximity snap** — if ground is not wired, `findNearestTerminal()` snaps it to the closest component pin by pixel distance.
8. **Electrical node naming** — iterate only over roots that component terminals actually belong to (skipping phantom union-find bookkeeping roots). Ground group → `"0"`, others → `"n1"`, `"n2"`, etc. This ensures the first real node is always `n1`, not `n3` or `n4`.
9. **Component ID** — uses `compNode.data.label` (the human-readable `V1`, `R1`, `C1`) with non-ASCII characters stripped to keep the SPICE file ASCII-safe.

**Error conditions thrown** (caught by SimulateButton, displayed inline):

- Circuit empty / no wires
- No ground present
- Component has wrong number of connections
- Both terminals at same node (direct short)
- Ground cannot be placed near any component
- Node `"0"` never assigned (ground not electrically connected)

### 3.5 Simulate Button (`SimulateButton.jsx`)

Handles the full simulation request lifecycle.

**Steps on click:**
1. Pre-flight checks: empty circuit, no wires → surface error inline (no `alert()`)
2. Call `convertCircuitToBackendFormat()` — catch and display converter errors
3. `POST /api/simulate` with the circuit JSON via axios
4. On success: call `onSimulate(response.data)` which triggers `App.jsx` to switch to the results page
5. On failure: call `onError(message)` which displays an inline error banner in the header

The button shows a CSS spinner animation while `isSimulating` is true. The button is `disabled` and `aria-busy` during the request.

### 3.6 Results Panel (`ResultsPanel.jsx`)

Displays the full simulation response. Used both in the sidebar (when results are shown inline) and in the full-screen `ResultsPage`.

**Sections rendered:**

**Node Voltages**  
Each voltage entry shows:
- Primary label: human-readable description built from `buildNodeDescriptions()` — e.g. `V1(+) · R1(A)` meaning "this node connects to V1's positive terminal and R1's pin A"
- Secondary label: the raw ngspice node name (`V(n1)`) as a monospace subtitle
- Value: formatted with `fmtVoltage()` — auto-selects V / mV / µV based on magnitude

**Branch Currents**  
Each current shows `I(V1)` mapped from the raw ngspice key (which may be the long ID) back to the clean component label via `buildCurrentLabel()`.  
Value formatted with `fmtCurrent()` — auto-selects A / mA / µA / nA.

**Structural Faults**  
A list of warning badges for any rule-based fault detected (yellow/amber styling).

**ML Fault Classification**  
A card showing:
- Predicted fault label (human-readable: "Value Drift", "Partial Short", etc.)
- Confidence badge — green ≥ 80%, amber 50–80%, red < 50%
- Card background — green for Normal, orange for any fault, grey for model unavailable
- Per-label probability bar chart showing all 4 fault class scores from the model

**Empty state:** Numbered steps instructing the user to build and simulate a circuit.

### 4.3 Circuit Validator (`validators.py`)

Runs before simulation to catch structurally invalid circuits early.

**`CircuitValidator.validate(circuit_data)`** returns `(is_valid, errors, warnings)`.

**Checks performed:**

| Check | Error or Warning |
|---|---|
| Required fields present (`nodes`, `components`, `ground`) | Error |
| At least one component | Error |
| At least one `dc_source` | Error |
| No duplicate component IDs | Error |
| Component type is known | Error |
| Component value within allowed range (e.g. resistor 1Ω–1MΩ) | Error |
| Component has the correct number of terminal nodes | Error |
| No self-loop (both terminals same node) | Error |
| Ground node exists in the nodes list | Error |
| Nodes declared but not used | Warning |
| Nodes used but not declared | Error |
| Nodes with only one component connection (floating) | Warning |

`ComponentSpec` stores the value ranges and terminal counts for all 7 component types. Measurement devices (ammeter, voltmeter) and ground skip value validation.


### 4.4 Netlist Generator (`netlist_generator.py`)

Converts the validated `circuit_dict` into a SPICE `.cir` file that ngspice can execute.

**Output format (example — simple RC circuit):**
```spice
* Circuit Fault Detector - Auto-generated Netlist
* Generated: 2026-07-13 15:42:00
* Analysis: DC Operating Point

V1 n1 0 DC 5.0
R1 n1 n2 1000
C1 n2 0 1e-07

* Control section
.control
  op
  print all
  print allv
  print alli
.endc

.end
```

**Component → SPICE mapping:**

| Type | SPICE line format |
|---|---|
| `dc_source` | `V{id} n+ n- DC {value}` |
| `resistor` | `R{id} n1 n2 {value}` |
| `capacitor` | `C{id} n1 n2 {value}` |
| `inductor` | `L{id} n1 n2 {value}` |
| `ammeter`, `voltmeter`, `ground` | skipped |

`_get_spice_name()` ensures the correct SPICE prefix is present — if the ID already starts with the right letter (e.g. `V1` for a dc_source), it is used as-is. Otherwise the prefix is prepended.

The `.control` block runs `.op` (DC operating point), then `print all`, `print allv`, `print alli` to force ngspice to output all node voltages and source currents to stdout.

### 4.5 Simulation Runner (`simulation_runner.py`)

Executes ngspice and parses its text output.

**Platform detection:** On Windows, uses `ngspice_con` (the console-mode build). On Linux/macOS, uses `ngspice`.

**Execution:**
1. Writes the netlist to a temp file at `{tmpdir}/circuit_fault_detector/circuit_{pid}.cir` with `encoding='utf-8'` (prevents charmap errors on Windows when component labels contain Unicode)
2. Runs `ngspice_con -b {netlist_file}` as a subprocess with a 10-second timeout
3. Deletes the temp file in `finally`

**Output parsing (`_parse_output`):**

ngspice produces two different output formats depending on platform and version. The parser handles both:

| Format | Example | Parsed as |
|---|---|---|
| Format 1 (Linux) | `v(n1) = 5.000000e+00` | Voltage at node n1 |
| Format 2 (Windows) | `n1 = 5.000000e+00` | Voltage at node n1 |
| Format 1 (current) | `i(v1) = -5.000000e-03` | Current through V1 |
| Format 2 (Windows) | `v1#branch = -5.000000e-03` | Current through V1 |

Special keywords (`allv`, `alli`, `all`, `temp`, `tnom`) are explicitly excluded from being parsed as node names.


### 4.6 Structural Fault Detector (`structural_faults.py`)

Runs after a successful ngspice simulation. Applies 8 rule-based checks and returns a list of human-readable fault strings.

Uses an internal **Union-Find** data structure for graph connectivity queries.

**Check 1 — Missing ground**  
Verifies that ground node `"0"` exists in the nodes list and at least one component is connected to it.

**Check 2 — Open circuits**  
Builds a union-find from all components (excluding voltmeters, which block DC). Any nodes not reachable from ground indicate a broken path. Also checks that every DC source has a path to ground.

**Check 3 — Short circuits**  
Uses ngspice results:
- Current > 1 A through any source → short circuit warning
- Resistor with near-zero voltage drop (< 0.01 V) while significant current flows → component shorted/bypassed
- Voltage source output < 10% of rated value while current flows → source is shorted

**Check 4 — Ammeter in parallel**  
An ammeter sharing both node connections with another component → should be in series (parallel placement creates a short).

**Check 5 — Voltmeter in series**  
Builds a second union-find excluding all voltmeters. If removing a voltmeter breaks the connectivity between its two nodes → it is in series (should be in parallel).

**Check 6 — Component bypasses**  
Checks if any resistor/capacitor/inductor has another very-low-resistance component (< 1 Ω) connecting the same two nodes — indicating an unintentional bypass.

**Check 7 — Reversed polarity**  
If the first node of a `dc_source` (conventionally positive) is the ground node `"0"` → the source is connected backwards.

**Check 8 — Series/parallel confusion**  
Currently a no-op placeholder. Would require design intent to implement meaningfully.

### 4.7 ML Fault Analyzer (`fault_analyzer.py`)

Loads the trained RandomForest model and runs inference on every simulation result.

**Model loading:** At import time, tries to import `joblib` and `pandas`. If either is missing, `ML_MODEL_AVAILABLE = False` and the analyzer returns an "unavailable" response gracefully. If all 4 model files exist and imports succeed, loads all artefacts on `__init__`.

**`analyze(circuit_data, node_voltages, branch_currents)`** — the main public method:

1. Extracts `component_values` dict (`{id: value}`) for resistors, capacitors, inductors, and dc_sources
2. Filters out the ground node from voltages (ground is always 0V by definition)
3. Calls `_extract_features()` to compute the 17-feature vector
4. Calls `_predict()` to run the RandomForest and interpret results

**Feature extraction** (must match `src/train.py` exactly):

| Feature | Description |
|---|---|
| `n_components` | Number of components with values |
| `comp_mean/max/min/std` | Statistics of component values |
| `n_nodes` | Number of non-ground signal nodes |
| `volt_mean/max/min` | Statistics of node voltages |
| `n_currents` | Number of measured branch currents |
| `curr_mean_abs/max_abs` | Statistics of absolute current values |
| `n_missing_currents` | Components minus currents (non-zero = capacitor swap) |
| `max_deviation_ratio` | Largest % deviation from nominal value |
| `second_deviation_ratio` | Second-largest % deviation |
| `deviation_ratio_2nd_over_1st` | Ratio of 2nd to 1st deviation (catches multi-faults) |
| `n_components_deviated_over_20pct` | Count of components > 20% off nominal |

**Deviation from nominal** requires the `nominal_lookup` — a dict keyed by `frozenset` of component IDs mapping to their average "normal" values from the training data. New circuit topologies not seen during training will have no nominal reference, so all deviation features will be 0.

**Prediction:** The model outputs per-label probabilities for 4 binary labels. Labels with probability ≥ 0.5 are "fired". The result is:
- No fired labels → `Normal`
- 1 fired label → that label (e.g. `drift`)
- 2+ fired labels → `Multiple_Faults (label1 + label2)`

## 5. ML Pipeline

### 5.1 Dataset Generator (`src/dataset_generator.py`)

Generates the training dataset by programmatically building, mutating, and simulating circuits.

**11 circuit topologies defined** (hardcoded, values derived from circuit theory where applicable — not guessed):

1. Series-parallel 4-resistor network
2. Multi-source 5-resistor network (KCL problem)
3. Single-resistor current source (P = I²R derived nominal)
4. 12k/9k voltage divider (Vs solved from given Vo)
5. KVL series loop ABCDEF (R values derived from given voltage drops at 1 mA)
6. VDR parallel network (current divider)
7. Wheatstone bridge (nominally balanced)
8. 3-stage R-ladder
9. Delta-loaded network
10. Two-source bridge
11. Resistor cube (12 edges, 8 nodes — R_eq = 5/6 Ω classic result)

**6 fault types × 20 samples × 11 circuits = 1,320 total samples**

**Fault types and how values are mutated:**

| Fault type | Mutation |
|---|---|
| `normal` | ±2% random variation on all resistors |
| `drift` | One resistor ±30–70% from nominal |
| `partial_short` | One resistor reduced to 1–15% of nominal |
| `partial_open` | One resistor increased to 5–50× nominal |
| `wrong_component_type` | One resistor replaced by a capacitor (1–10 µF) |
| `multi_fault` | Two different fault types applied to two different components |

For each sample, `dataset_generator.py`:
1. Applies the mutation to the nominal values
2. Writes a `.cir` netlist (saved under `output/circuits/{circuit_name}/`)
3. Runs `ngspice_con -b` on it
4. Parses node voltages and branch currents
5. Records all data as a CSV row in `output/manifest.csv`

The final combined dataset is saved to `dataset/dataset.csv`.

---

### 5.2 Training (`src/train.py`)

Trains a **multi-label RandomForest classifier** on the dataset.

**Multi-label formulation:** Instead of one multi-class label per sample, each sample has 4 independent binary labels (`drift`, `partial_short`, `partial_open`, `wrong_component_type`). A `multi_fault` sample can have 2 labels set to 1 simultaneously.

**Nominal lookup construction:** Computes per-circuit-topology average values for all "normal" samples. Used to compute deviation-from-nominal features.

**Training:**
- 70/30 train/test split
- `RandomForestClassifier(n_estimators=200, class_weight="balanced")`
- `class_weight="balanced"` compensates for label imbalance

**Evaluation metrics reported:**
- Per-label classification report (precision, recall, F1)
- Hamming loss (fraction of labels predicted wrong)
- Exact-match accuracy (all 4 labels correct simultaneously)

**Saved artefacts:**

| File | Contents |
|---|---|
| `models/fault_classifier.joblib` | Trained RandomForest (200 trees, multi-output) |
| `models/feature_columns.joblib` | Ordered list of 17 feature names |
| `models/label_columns.joblib` | `['drift', 'partial_short', 'partial_open', 'wrong_component_type']` |
| `models/nominal_lookup.joblib` | `{frozenset(component_ids): {id: mean_value}}` |

---

### 5.3 Predictor (`src/predictor.py`)

A standalone module for running predictions outside the API — used by test scripts like `generate_test_sample.py`.

`predict_one(sample)` takes a dict with JSON strings for `component_values`, `node_voltages`, `branch_currents` — the same schema as the dataset CSV rows.

`register_nominal(component_names, nominal_values)` allows injecting nominal values for circuit topologies not in the training set.

## 6. End-to-End Data Flow

```
USER draws circuit on canvas
         │
         ▼
CircuitCanvas.jsx
  ReactFlow nodes + edges → setCircuit({ nodes, edges })
         │
         ▼
SimulateButton.jsx  [▶ Simulate clicked]
         │
         ▼
circuitConverter.js  convertCircuitToBackendFormat()
  • Classifies nodes (components / junctions / ground)
  • Builds adjacency graph from edges
  • Validates per-pin wire counts
  • Union-Find: merges terminals, junctions, ground into electrical nodes
  • Assigns n1, n2, ... (only real terminal roots, no phantom counters)
  • Returns { nodes[], components[{id, type, value, nodes[]}], ground:"0" }
         │
         ▼
POST /api/simulate  (JSON body)
         │
         ▼  backend/main.py
Step 1: CircuitValidator.validate()
  • Checks types, value ranges, terminal counts, duplicate IDs
  • Returns errors[] + warnings[]
         │
         ▼
Step 2: NetlistGenerator.generate()
  • Maps each component to a SPICE line
  • Adds .control / op / print all / .endc block
  • Returns netlist string
         │
         ▼
Step 3: SimulationRunner.run_simulation()
  • Writes netlist to temp .cir file (UTF-8)
  • Runs ngspice_con -b {file}  (10s timeout)
  • Parses stdout for v(node) and i(source) patterns
  • Returns { success, voltages{}, currents{} }
         │
         ▼
Step 4: StructuralFaultDetector.detect_faults()
  • 7 active rule checks (open, short, reversed polarity, etc.)
  • Returns List[str] of fault descriptions
         │
         ▼
Step 5: FaultAnalyzer.analyze()
  • Builds component_values dict from circuit definition
  • Extracts 17 features (component stats, voltage stats, deviation-from-nominal)
  • Runs RandomForest.predict_proba() → per-label probabilities
  • Fires labels above 0.5 threshold
  • Returns { predicted_fault, confidence, all_probabilities, description }
         │
         ▼
SimulationResponse returned as JSON
  { success, netlist, structural_faults[], pattern_faults{}, simulation_data{} }
         │
         ▼
App.jsx  handleSimulateResults()
  • Stores results, sets showResultsPage=true
         │
         ▼
ResultsPage.jsx + ResultsPanel.jsx
  • Shows read-only canvas replay
  • Renders voltages with human-readable node labels
  • Renders currents with clean component IDs
  • Shows structural fault list
  • Shows ML card with confidence badge + probability bar chart
```

## 7. API Reference

### `GET /`
Health check. Returns `{ status: "online", service, version }`.

### `GET /api/health`
Detailed status. Returns ngspice installation status + version, and whether the ML model is loaded.

### `GET /api/components`
Returns the list of all supported component types with value ranges, units, and descriptions.

### `POST /api/simulate`
Main simulation endpoint.

**Request body:** `CircuitModel` JSON  
**Response:** `SimulationResponse` JSON

Returns `success: false` with error details if validation fails or ngspice returns a non-zero exit code. Always returns `structural_faults` and `pattern_faults` on success even if they are empty.

### `POST /api/validate`
Validates circuit structure without running simulation. Returns `{ valid, errors[], warnings[] }`.

## 8. Known Limitations & Open Issues

### Circuit Editor

| Issue | Detail |
|---|---|
